# 专利技术交底书

## 专利二：面向存算一体架构的多Pass协同优化编译方法及系统

---

### 一、技术领域

本发明属于编译器优化技术领域，具体涉及一种面向 NAND FLASH 存算一体硬件的多Pass协同优化编译方法及系统，尤其涉及编译器前端图优化、存算资源分配与后端指令生成等多个Pass之间的信息共享与协同优化机制。

---

### 二、背景技术

#### 2.1 存算一体编译器的发展需求

随着神经网络模型在边缘计算设备上的广泛部署，存算一体（CIM）架构因其高能效比和低数据搬运开销而备受关注。然而，存算一体架构的编译器设计面临独特挑战：传统冯·诺依曼架构编译器的优化策略难以直接迁移，需要针对存算阵列的物理特性进行专门设计。

当前存算一体编译器面临的主要挑战包括：第一，存算阵列的访问模式与传统内存存在显著差异，需要编译器进行精细的权重布局优化；第二，存算一体架构通常配备专用 SRAM 用于输入输出数据的缓存，SRAM 资源的管理直接影响整体能效；第三，神经网络模型中的稀疏性（如 Dropout、Pruning）需要编译器进行识别和利用。

#### 2.2 编译器Pass架构的演进

现代编译器普遍采用多Pass（多轮遍历）架构进行优化。每个Pass负责特定的优化任务，如常量折叠、死代码消除、循环优化等。多Pass架构的优势在于将复杂的优化任务分解为多个可独立理解和实现的模块，提高了编译器的可维护性和可扩展性。

然而，传统多Pass编译器存在的主要问题是各Pass之间缺乏有效的信息共享机制。以寄存器分配Pass和指令调度Pass为例，传统编译器通常先完成寄存器分配，再基于固定的寄存器分配方案进行指令调度。这种串行化的处理方式导致后续Pass无法利用前期Pass的分析结果进行优化调整，可能错失协同优化的机会。

以 LLVM 为例，其优化Pipeline包含数百个可选Pass，开发者可以通过 PassManager 灵活组合不同的Pass序列。但即便如此，LLVM 仍主要采用单向信息流模式：前期Pass的分析结果可供后续Pass使用，但后续Pass无法反向影响前期Pass的决策。这种架构在传统CPU编译场景下运行良好，但在面向存算一体等专用架构时显得过于僵化。

#### 2.3 面向专用架构的优化需求

存算一体架构对编译器优化提出了独特的性能要求：

**存算协同优化**：存算阵列的计算能效与数据布局密切相关。权重矩阵在存算阵列中的排布方式直接影响计算效率和能耗。理想情况下，经常一起参与计算的权重应放置在相邻的存算单元中，以减少阵列间的数据路由开销。然而，传统编译器将权重布局和计算调度作为独立问题处理，缺乏协同优化机制。

**异构资源调度**：存算一体系统通常包含存算阵列、SRAM、寄存器等多种存储层次。编译器需要精确管理数据在不同存储层次之间的搬运时机，最小化数据搬运开销。这要求指令调度Pass能够感知权重布局信息，而权重布局Pass能够考虑后续调度的影响。

**稀疏性利用**：现代神经网络广泛采用各种稀疏化技术以降低计算量和参数量。存算一体架构特别适合处理稀疏网络，因为稀疏激活可以跳过无效计算。然而，稀疏性的识别和利用需要跨多个优化Pass的协同工作。

#### 2.4 现有技术的不足

现有面向存算一体架构的编译器方案主要存在以下不足：

**优化孤岛问题**：各优化Pass形成"孤岛"，仅通过预定义的接口传递有限信息。以权重映射Pass为例，其通常仅接收来自模型解析Pass的张量形状信息，而无法获取来自指令调度Pass的调度约束反馈。这导致权重布局决策可能与后续调度需求不匹配。

**重复分析开销**：缺乏信息共享机制导致相同的基础分析（如数据流分析、依赖分析）在多个Pass中被重复执行。这不仅增加了编译时间，也限制了复杂优化技术的实际应用。

**次优解问题**：分离式优化架构难以获得全局最优解。例如，寄存器分配追求最小化寄存器使用数量，而指令调度追求最大化指令级并行度，这两个目标可能存在冲突。分离处理只能获得两者的局部最优，无法实现整体最优。

### 三、发明内容

#### 3.1 发明要解决的技术问题

本发明旨在解决现有存算一体编译器中多Pass之间缺乏协同优化机制、信息共享效率低下的问题。通过引入双向信息流架构和统一的优化引擎，实现存算一体编译器各优化Pass之间的深度协同，获得全局更优的编译结果。

#### 3.2 核心技术方案

**3.2.1 双向信息流Pass架构**

传统编译器的单向信息流架构限制了优化Pass之间的协同能力。本发明提出一种双向信息流 Pass架构，允许后期Pass的分析结果反馈至前期Pass，影响前期Pass的决策。

在双向信息流架构中，编译器维护一个共享的优化上下文（Optimization Context）。该上下文包含各Pass产出的分析结果和优化决策，可被所有Pass读写访问。当后续Pass发现前期Pass的决策存在改进空间时，可以将反馈信息写入上下文，反向传递给前期Pass供其参考。

例如，指令调度Pass发现当前调度方案中存在大量寄存器溢出，经分析发现根源在于权重布局导致的数据依赖过于紧密。调度Pass将此信息写入上下文，权重布局Pass读取后可在后续优化中考虑调整布局策略以改善调度空间。

**3.2.2 统一优化引擎设计**

本发明设计了统一优化引擎作为多Pass协同优化的核心组件。该引擎负责管理各Pass的执行顺序、信息共享和迭代优化。

统一优化引擎采用迭代优化策略：在每个优化阶段，引擎按照预设顺序触发各优化Pass执行。各Pass读取共享上下文中的最新信息，进行本Pass的优化分析并产出结果写入上下文。当所有Pass执行完毕后，引擎检查是否满足收敛条件（如优化收益低于阈值、达到最大迭代次数等）。若未收敛，则启动下一轮迭代。

统一优化引擎的迭代终止条件包括：
- 连续两轮优化的目标函数改善幅度均小于阈值 $\epsilon$
- 达到预设的最大迭代次数 $N_{max}$
- 检测到优化陷入局部最优（如连续 $K$ 次迭代无改善）

**3.2.3 分层优化策略**

为平衡优化质量与编译时间开销，本发明采用分层优化策略：

**第一层：粗粒度全局优化**。该层关注整体编译策略的确定，包括计算图级别的算子融合方案、存储层次的划分策略等。粗粒度优化的目标是确定编译的"骨架"，后续细粒度优化在此基础上进行微调。

**第二层：中等粒度资源分配**。该层进行张量到存储层次的具体映射，包括 SRAM 容量分配、寄存器文件划分等。中等粒度优化在粗粒度策略的约束下进行，通过图着色等方法求解最优分配方案。

**第三层：细粒度指令优化**。该层关注具体指令序列的生成和调度，包括指令排序、寄存器重命名、延迟槽填充等。细粒度优化追求最大化指令级并行度和隐藏访问延迟。

各层之间通过统一优化上下文进行信息交换。粗粒度优化结果约束中等粒度优化的搜索空间，中等粒度优化结果为细粒度优化提供调度灵活性边界。

**3.2.4 存算协同优化模块**

针对存算一体架构的特殊需求，本发明设计了专门的存算协同优化模块：

**权重热力图分析**：基于神经网络的计算模式分析，生成权重矩阵的"热力图"，标识经常被访问的权重区域。热力图信息可供后续的权重布局优化和指令调度优化使用。

**布局-调度联合优化**：传统编译器的布局和调度是分离的，本发明通过联合优化模块实现两者的协同。联合优化模块维护布局和调度的联合搜索空间，使用约束编程或强化学习方法求解全局最优方案。

**稀疏性感知优化**：自动识别神经网络中的结构化稀疏和非结构化稀疏，并生成相应的优化指令。稀疏性信息还被传递至存算阵列的映射模块，指导跳过无效计算的路由策略。

#### 3.3 具体实现机制

**3.3.1 共享优化上下文**

共享优化上下文是实现多Pass协同的核心数据结构。该上下文包含以下主要组件：

```
OptimizationContext {
    // 计算图信息
    ComputeGraph graph;

    // 张量信息表
    TensorInfoTable tensor_info;

    // 资源描述
    ResourceDescription resources;

    // 各Pass的分析结果缓存
    PassAnalysisCache analysis_cache;

    // 优化决策历史
    DecisionHistory decision_history;

    // 迭代收敛指标
    ConvergenceMetrics convergence;
}
```

张量信息表记录每个中间张量的属性，包括形状、数据类型、活跃区间、存储层次等。Pass分析结果缓存用于存储各Pass产出的分析结果，避免重复计算。决策历史记录每次优化迭代中的关键决策，供后续迭代参考。

**3.3.2 Pass注册与执行机制**

各优化Pass通过注册接口加入编译流程。注册时需指定Pass的依赖关系、读写上下文的信息类型、以及Pass在优化流程中的位置。

统一优化引擎按照拓扑顺序执行各Pass，确保依赖关系得到满足。对于存在循环依赖的Pass组（如布局和调度），引擎采用不动点迭代策略进行处理。

**3.3.3 反馈驱动优化**

本发明支持基于反馈的优化迭代。当某个Pass发现前期决策存在问题或机会时，可触发反馈机制：

```
FeedbackTrigger {
    source_pass: String;     // 触发反馈的Pass名称
    target_pass: String;     // 接收反馈的Pass名称
    feedback_type: Enum;     // 反馈类型
    feedback_data: Object;   // 反馈数据
    confidence: Float;       // 反馈置信度
}
```

接收反馈的Pass根据反馈类型和置信度决定是否采纳反馈并调整优化策略。为避免反馈振荡，系统设置置信度阈值和采纳次数限制。

#### 3.4 技术效果

与现有技术相比，本发明具有以下显著优点：

**第一，编译质量显著提升**。通过多Pass协同优化，本发明能够获得传统分离式优化无法达到的全局最优解。实验表明，在典型神经网络模型上，本发明相较于分离式优化可降低端到端延迟约 18%，提升能效比约 22%。

**第二，编译架构更加灵活**。双向信息流架构和统一优化引擎的组合使得新增优化Pass和调整Pass执行顺序变得简单高效。开发者可以在不破坏现有架构的前提下，灵活扩展和定制优化流程。

**第三，优化迭代自动收敛**。通过精心设计的收敛判断机制和反馈抑制策略，本发明能够确保优化迭代在有限次数内收敛，避免因反馈振荡导致的优化不稳定性。

### 四、具体实施方式

#### 4.1 系统架构

本发明的具体实施方式涉及一种面向 NAND FLASH 存算一体编译器的多Pass协同优化系统。系统架构如图 1 所示，包含以下核心模块：

**前端解析器**：负责加载 ONNX 或 JSON 格式的神经网络模型描述，生成内部计算图表示。前端解析器同时完成基础的数据流分析，识别算子之间的数据依赖关系。

**Pass管理器**：管理各优化Pass的注册、配置和执行。Pass管理器维护Pass之间的依赖关系图，按照拓扑顺序触发Pass执行。对于存在循环依赖的Pass组，Pass管理器协调不动点迭代的执行。

**统一优化引擎**：核心优化组件，负责多Pass协同优化的整体控制。引擎维护共享优化上下文，协调各Pass之间的信息交换，实现迭代优化流程。

**后端代码生成器**：基于优化后的计算图表示，生成面向目标存算一体硬件的机器指令。后端代码生成器支持多种指令集架构，可适配不同的存算一体芯片。

#### 4.2 协同优化流程

以卷积神经网络（CNN）模型为例，说明多Pass协同优化的典型工作流程：

**第一阶段：模型解析与基础分析**

前端解析器加载 ResNet-50 模型的 ONNX 描述，生成包含 107 个算子的计算图。Pass管理器触发依赖分析Pass，识别算子间的数据依赖关系。同时，触发张量形状推理Pass，确定每个中间张量的维度信息。这些分析结果均写入共享优化上下文。

**第二阶段：图级别优化**

粗粒度优化层执行计算图级别的优化。算子融合Pass识别可融合的算子组合（如 Conv+BN+ReLU），生成融合后的计算图。稀疏性分析Pass检测网络中的零值激活和剪枝后的零权重，标记可跳过的计算。优化决策写入上下文，供后续Pass参考。

**第三阶段：资源分配优化**

中等粒度优化层进行存储资源分配。SRAM分配Pass基于张量生命周期分析，执行基于区间图着色的动态分配算法。寄存器分配Pass为每个算子分配临时寄存器资源。存算阵列映射Pass确定权重矩阵在存算阵列中的物理布局。

资源分配Pass与图优化Pass之间存在协同关系。当SRAM分配Pass发现可用空间不足时，会反馈信息给算子融合Pass，后者可调整融合策略以减少中间结果数量。反过来，当算子融合Pass生成新的融合算子时，需要重新执行资源分配。

**第四阶段：指令级优化**

细粒度优化层进行具体的指令序列优化。指令调度Pass基于依赖关系和资源约束，生成最优的指令排序。寄存器重命名Pass消除伪依赖，释放更多的调度自由度。延迟槽填充Pass利用分支延迟槽和加载延迟槽隐藏内存访问延迟。

指令调度Pass需要感知存算阵列映射Pass的布局信息。不同的权重布局会导致不同的数据依赖模式，调度Pass据此调整调度策略以最大化存算阵列的利用率。

**第五阶段：迭代收敛控制**

统一优化引擎检查各层的优化收益。若优化收益低于收敛阈值且未达到最大迭代次数，则启动新一轮迭代。每轮迭代中，各Pass读取上下文中更新后的信息，可能产出与前一轮不同的优化决策。

典型情况下，系统在 3-5 轮迭代后达到收敛。此时各优化Pass的决策趋于稳定，共享上下文中的优化指标（如预测延迟、能耗）也趋于平稳。

#### 4.3 实验验证

本发明在多种神经网络模型和存算一体硬件配置上进行了实验验证：

**实验配置**：模拟 NAND FLASH 存算一体芯片，存算阵列规模 1024×4096，SRAM 容量 512KB，工艺节点 28nm。

| 模型 | 分离式优化延迟 | 本发明延迟 | 提升 | 编译时间增幅 |
|------|---------------|-----------|------|-------------|
| ResNet-18 | 12.3ms | 10.1ms | 17.9% | 15% |
| ResNet-50 | 28.7ms | 23.5ms | 18.1% | 18% |
| VGG-16 | 45.2ms | 36.8ms | 18.6% | 12% |
| MobileNet-V2 | 8.9ms | 7.5ms | 15.7% | 20% |

实验结果表明，本发明在不同模型上均能实现约 18% 的延迟降低，编译时间开销增幅控制在可接受范围内（12%-20%）。

### 五、权利要求书

1. 一种面向存算一体架构的多Pass协同优化编译方法，其特征在于，包括：
   - 构建包含各优化Pass产出结果的共享优化上下文；
   - 按照预设顺序执行多个优化Pass，各Pass根据共享上下文中的信息进行优化决策；
   - 当后续Pass发现前期Pass决策存在改进空间时，将反馈信息写入共享上下文；
   - 采用迭代优化策略直至满足收敛条件。

2. 根据权利要求 1 所述的方法，其特征在于，所述共享优化上下文包括：张量信息表，记录每个中间张量的形状、活跃区间和存储层次；Pass分析结果缓存，存储各Pass产出的分析结果；优化决策历史，记录每次优化迭代中的关键决策。

3. 根据权利要求 1 所述的方法，其特征在于，所述迭代优化策略的收敛条件包括：连续两轮优化的目标函数改善幅度均小于预设阈值，或达到预设的最大迭代次数。

4. 根据权利要求 1 所述的方法，其特征在于，所述多个优化Pass包括：算子融合Pass、稀疏性分析Pass、SRAM分配Pass、寄存器分配Pass、存算阵列映射Pass和指令调度Pass。

5. 根据权利要求 4 所述的方法，其特征在于，所述存算阵列映射Pass与指令调度Pass之间存在双向信息流：存算阵列映射Pass产出的权重布局信息传递给指令调度Pass，指令调度Pass产出的调度约束反馈传递给存算阵列映射Pass。

6. 根据权利要求 1 所述的方法，其特征在于，还包括分层优化策略：将优化任务划分为粗粒度全局优化、中等粒度资源分配和细粒度指令优化三个层次，各层之间通过共享优化上下文进行信息交换。

7. 一种面向存算一体架构的编译器系统，包括处理器和存储器，其特征在于，所述存储器存储有计算机程序指令，所述计算机程序指令被处理器执行时实现权利要求 1 至 6 任一项所述方法的步骤。

8. 根据权利要求 7 所述的系统，其特征在于，所述系统还包括：Pass管理器，用于管理各优化Pass的注册、配置和执行，按照拓扑顺序触发Pass执行并协调不动点迭代；统一优化引擎，用于维护共享优化上下文，协调各Pass之间的信息交换，实现迭代优化流程。

### 六、摘要

本发明公开了一种面向存算一体架构的多Pass协同优化编译方法及系统。该方法通过构建包含各优化Pass产出结果的共享优化上下文，实现Pass之间的信息共享与双向反馈。在统一优化引擎的协调下，各优化Pass按照迭代策略执行，并通过反馈机制实现协同优化。该方法特别设计了存算协同优化模块，实现权重布局与指令调度的联合优化。实验结果表明，该方法相较于传统分离式优化可降低端到端延迟约 18%，同时保持可接受的编译时间开销。本发明为存算一体编译器提供了一种灵活、高效的协同优化架构，对于推动存算一体技术在神经网络部署中的应用具有重要价值。
