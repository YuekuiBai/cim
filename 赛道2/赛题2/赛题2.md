# 赛道二：编译工具链 - 赛题二

## 赛题背景及意义

随着 Llama 70B、Mixture-of-Experts (MoE) 等大语言模型的爆发，模型参数量已突破千亿级别，传统的计算架构面临着严峻的"**存储墙**"与"**功耗墙**"挑战。

**存算一体 (Computing-in-Memory, CIM)** 技术通过将计算融入存储介质（如 3D NAND Flash），实现了"**位置即算力**"的革新。然而，这种架构引入了一个全新的数学挑战：**如何在极度受限的 3D 物理空间中，高效地排布和调度巨大的神经网络模型？**

本赛题要求参赛者设计一个**智能编译器/调度器**，将复杂的深度学习模型（计算图）映射到一个抽象的**分层 3D 异构资源空间**中。这不仅是一个内存分配问题，更是一个涉及并行计算、流水线优化和异构调度的系统级难题。

## 赛题任务

### 2.1 赛题任务流程

参赛者需搭建从神经网络算法到硬件部署的简化流程框架，包括但不限于以下三个环节：

#### a. 模型解析

| 项目 | 内容 |
|------|------|
| **输入** | ONNX 格式的神经网络模型 |
| **任务** | 完成算子（如 MatMul, MoE Experts）的参数提取，根据给定的规则，确定待部署的**权重立方块 (Weight-Cube)** 的参数 |

#### b. 编译映射

| 项目 | 内容 |
|------|------|
| **任务** | 将"存算阵列"权重映射排布抽象为数学建模问题 |
| **抽象表示** | - 使用 **"存算核 (Sub-Cube)"** 表示存算阵列资源<br>- 使用 **"Weight-Cube"** 表示一个权重块及其放置区域 |
| **核心算法** | 设计高效算法，完成"Weight-Cube"在"存算核"范围内的**无重叠排布** |
| **优化目标** | 针对时延、空间利用率等指标优化算法 |

#### c. 配置验证

| 项目 | 内容 |
|------|------|
| **任务** | 设计模拟器完成结果的测试验证 |
| **验证内容** | **1. 测试排布的合理性**：检查是否违反空间互斥等物理约束<br>**2. 测试存算阵列的执行效率** |

**并行性规则：**

| 规则类型 | 说明 |
|---------|------|
| **Sub-Cube 间并行 (Inter-Cube Parallelism)** | 不同的 Sub-Cube 可以完全并行工作 |
| **Sub-Cube 内互斥 (Intra-Cube Exclusivity)** | 每个 Sub-Cube 在同一时刻只能激活唯一的一个 Weight-Cube |

**并行冲突惩罚：**

- 若将逻辑上需要**同时并行计算**的两个 Weight-Cube（如 Shared Attention 和 Shared MLP）映射到了**同一个 Sub-Cube**中，硬件将无法同时激活它们
- 模拟器将强制对其进行**串行调度**，导致端到端推理延时显著增加

#### 1. 工程实现规范

| 要求 | 说明 |

#### 2. 编译映射环节要求

参赛者需定义完整的 Cube 配置，包括：

| 配置项 | 约束条件 |

| **划分使用** | 允许对权重立方体进行切分，但仅限**水平**或**垂直**方向的切分，以适配定义的阵列规模 |

### 2.3 问题定义

#### 硬件模型：3D 资源空间 (3D Resource Space)

<div align="center">
  <img src="图5.png" width="500" alt="3D 资源空间"/>
  <p>图 5 3D 资源空间</p>
</div>

参赛者需要先定义整个 Cube 的配置（包括 N×N、D×W×H），然后在此空间内进行权重映射和调度。
我们将硬件抽象为一个三级层级的 3D 阵列结构：

**1. 全局空间 (Global Space)：**

| 特性 | 说明 |
|------|------|
| **空间划分** | 整个硬件资源被划分为 N×N 个相互独立的"存算核"(Sub-Cubes) |
| **并行特性** | 所有 Sub-Cubes 可以同时工作，但同时计算并行 Sub-Cube 数量有上限，彼此互不干扰。这提供了宏观的并行度（类似于 GPU 的 SM 或 CPU 的 Core） |

**2. 存算核 (Sub-Cube)：**

| 特性 | 说明 |
|------|------|
| **内部结构** | 每个 Sub-Cube 内部可以放置多个"权重立方体"(Weight-Cube) |
| **互斥特性** | 在任意时刻，每个 Sub-Cube 内部只能激活唯一的一个 Weight-Cube |

**3. 权重立方体 (Weight-Cube)：**

| 特性 | 说明 |
|------|------|
| **定义** | Weight-Cube 是神经网络权重矩阵（或其切分后的部分）在 3D 空间中的映射单元 |
| **放置规则** | 一个 Sub-Cube 内可放置多个不重叠的 Weight-Cube |
| **激活规则** | 每次计算只能激活一个 Weight-Cube，消耗 **1 + Weight-Cube 的 D** 个周期，即执行一次矩阵运算 |

---

#### 输入数据：模型计算流 (Model Computation Stream)

输入包含两部分：

**a. 模型描述文件 (model.json)**

经过预处理的神经网络计算流，包含一系列**算子 (Operators)**：

| 属性 | 说明 |
|------|------|
| **权重 (Weight)** | 每个算子包含一个或多个权重矩阵（如 Linear 层的 W），需要由参赛者根据规则切分为适配存算核 (Sub-Cube) 大小的权重立方体 |
| **依赖关系 (Dependencies)** | 算子之间存在数据依赖（如 Layer N 的输出是 Layer N+1 的输入） |
| **稀疏特性 (Sparsity)** | 对于 MoE 模型，某些算子（Experts）是条件激活的，具有稀疏性 |
| **权重操作 (Weight Operations)** | **复制 (Replication)**：允许将同一个权重矩阵立方体（Weight-Cube）复制多份，放置在不同的 Sub-Cubes 中（支持数据并行）<br>**切分 (Partitioning)**：允许将大权重矩阵沿水平或垂直方向切分为多个小块（Section），分散映射到不同位置（支持模型并行） |

**b. 激活轨迹文件 (activation_trace.json)【MoE 模型必需】**

记录 MoE 模型的真实 Expert 激活模式，用于互斥分析和分组优化：

```json
{
  "num_experts": 256,
  "top_k": 8,
  "shared_experts": [0],
  "n_inferences": 10000,
  "traces": [
    [0, 15, 33, 48, 72, 128, 200, 255],
    [0, 8, 42, 56, 99, 150, 201, 240],
    ...
  ]
}
```

**关键用途：**

| 用途 | 说明 |
|------|------|
| **1. 构建共现矩阵** | 统计任意两个 Expert 同时激活的次数 |
| **2. 识别互斥关系** | 共现次数为 0 的 Expert 对可共享 Sub-Cube |
| **3. 计算激活频率** | 高频 Expert 应放置在低成本位置（z=0） |
| **4. 优化分组策略** | 最小化 Sub-Cube 内部的冲突概率 |

---

#### 输出目标：空间映射与时间调度

参赛者需要生成一个**解决方案 (Solution)**，包含：

| 输出 | 内容 |
|------|------|
| **1. 空间映射 (Spatial Mapping)** | 决定每个权重立方体 (Weight-Cube) 放置在哪个 Sub-Cube 上 |
| **2. 静态调度 (Static Scheduling)** | 基于数据流依赖，确定计算的逻辑执行顺序 |

---

#### 核心约束

解决方案必须满足以下硬性物理约束：

**1. 物理互斥与激活规则**

| 规则 | 说明 |
|------|------|
| **Weight Stationary (权重驻留)** | 权重立方体 (Weight-Cube) 一旦映射至物理位置，在推理运行期间不可动态迁移 |

**2. 最大并行约束**

| 规则 | 说明 |
|------|------|
| **One Weight-Cube Per Sub-Cube** | 基于 3D 堆叠特性，单个 Sub-Cube 在同一时刻仅允许激活 1 个 Weight-Cube |
| **约束公式** | $\forall t, \forall s \in SubCubes: \|Active(s, t)\| \leq 1$ |

**3. 时序同步与切换代价**

| 规则 | 说明 |
|------|------|
| **Strict Dependency (严格依赖)** | 遵循层级流水线约束，下游 Layer 必须等待上游 Layer 所有 Section 输出就绪方可激活（Barrier Synchronization），否则产生流水线气泡 |
| **Switching Penalty (切换代价)** | **Intra-Sub-Cube (Weight-Cube Switching)**：高代价。同一 Sub-Cube 内切换激活不同 Weight-Cube，产生 **D** 周期的延迟 |

---

#### 优化目标

参赛者的算法应在满足上述约束的前提下，优化以下指标（按优先级排序）：

| 优先级 | 指标 | 说明 |
|:------:|------|------|
| 1 | **端到端推理延时 (End-to-End Latency)** | 完成整个模型推理所需的总周期数。这是并行度和流水线效率的综合体现 |
| 2 | **空间利用率 (Space Utilization)** | 成功映射的模型参数总量与占用 3D 空间体积的比值 |
| 3 | **大模型适配性 (LLM Capability)** | 能够成功映射 DeepSeek-671B 模型（在给定的 3D 空间下） |

## 赛题场景

### DeepSeek-671B (MoE) 的细粒度专家

**DeepSeek-V3** 拥有 **671B 总参数**，采用 **Fine-Grained MoE 架构**（256 个路由专家，每次激活 8 个路由专家和一个共享专家）。

**架构特性：**

| 特性 | 挑战 |
|------|------|
| **Expert 数量极多但单个 Expert 较小** | 需要精细的空间划分策略 |
| **Top-K 路由机制** | 专家激活呈现高度稀疏性 |
| **互斥性与共现性** | 某些专家极少同时被选中，而特定的专家组则频繁协同工作 |
| **共享专家** | 始终激活的"热点"与访问频率呈长尾分布的路由专家形成了鲜明的冷热数据差异 |
| **并发处理需求** | 推理过程要求并发处理多个被选中的专家，这种动态的并行计算需求使得共现专家若发生资源抢占将直接导致严重的性能瓶颈 |

这是典型的 **3D CIM 部署挑战场景**。

---

## 核心挑战

| 挑战 | 详细说明 |
|------|---------|
| **1. 参数量巨大** | 671B 参数需要高效利用整个 3D 空间（N 个 Sub-Cube × D 层 × H×W 尺寸） |
| **2. 稀疏激活** | 每次推理仅激活部分 Expert，需要合理利用不同领域专家激活频率特性 |
| **3. 混合结构** | Shared Experts（始终激活）+ Routed Experts（条件激活） |
